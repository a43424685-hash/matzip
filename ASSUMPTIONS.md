# 구현 중 내린 가정 / 판단 (Assumptions)

기획 논의를 최소화하고 합리적으로 가정해 바로 구현했습니다. 명세 20장의 "아직 결정이 필요한 질문"과 모호했던 지점에 대해 내린 결정을 정리합니다. 모두 나중에 쉽게 바꿀 수 있게 설계했습니다.

## 플랫폼 / 스택
1. **첫 출시 플랫폼: 웹/PWA 우선.** Next.js App Router. 모바일 우선 레이아웃(폰 너비 컬럼 + 하단 탭).
2. **DB: Supabase Postgres.** Prisma + `prisma db push`로 스키마 동기화(마이그레이션 폴더 없음). enum은 `String + 주석 허용값` 유지. (초기 SQLite 가정에서 전환 완료.)
3. **회원가입: 이메일 + 비밀번호(bcrypt) + HMAC 서명 쿠키 세션 + 카카오 OAuth.** 이메일 가입은 인증메일 없이 즉시 사용(emailVerifiedAt 즉시 세팅). 카카오는 이메일 인증여부 확인 후 병합. NextAuth 미도입(의존성 최소화).
4. **지도/주소: 카카오맵 SDK + 주소검색 연동.** 등록 시 카카오 장소검색으로 좌표·주소 확보, 상세/주변에 카카오맵 표시. 위치 인증은 GPS 50m 기준.

## 미디어 / 영상
5. **사진/영상은 Supabase Storage 업로드.** 등록 폼에서 파일 업로드 → Storage URL 저장. `Media.type = image|video`, `thumbnailUrl`, `duration` 컬럼 보유. 영상은 카드/상세에서 `<video>`로 재생. (초기 URL 입력 방식에서 전환 완료.)
6. **릴스형 전체화면 숏폼 피드는 미구현.** 일반 카드에서 영상 표시까지만.

## XP / 레벨 / 어뷰징 (명세 6장 기준)
7. **등록 XP 합계 예시**: 사진+한줄평+상세리뷰+카테고리3+가격+재방문 = `100(등록)+30+40+80+40+20+20 = 330`, 여기에 "하루 첫 등록" 100 보너스가 붙으면 430. 명세의 "한 곳 제대로 등록 시 250~450 XP" 범위와 일치.
8. **메뉴 추천 입력(+30 XP)**: 등록 폼에 필드를 아직 두지 않아 미지급. `sourceType: menu_recommend`는 `xpRules`에 정의되어 있어 폼 필드만 추가하면 동작.
9. **반응 XP 귀속**: 좋아요/저장 받은 XP는 **글 작성자**에게, 해당 음식점 **primary region**에 적립(전체 XP도 동시 증가).
10. **저장 XP**: "동일 사용자→동일 음식점 1회만" 작성자에게 +25. 저장은 restaurant 단위(명세 saves 구조)라 음식점 기준 멱등.
11. **좋아요 멱등 정책(명세 18.2 채택)**: 최초 좋아요에만 XP, 취소 후 재좋아요는 재지급 없음(`dedupeKey`로 보장).
12. **어뷰징 한도 기본값**: 하루 등록 기본 XP 최대 10건 / 같은 사람→같은 사람 좋아요 XP 하루 3건(감쇠) / 셀프 좋아요·저장 XP 없음 / 동일 음식점(동명+동지역) 중복 등록 기본 XP 없음. "신규 계정 좋아요 지연 반영", "신고 누적 보류"는 데이터 구조만 두고 로직은 향후.
13. **장기 활동 XP**: "하루 첫 등록(+100)"만 구현. 7일 연속/지역 10곳/카테고리 10곳/무신고 30일은 배치·집계가 필요해 `sourceType`만 정의하고 미구현.
14. **레벨 공식**: 명세 그대로 `300 + floor(level^2.15 * 90)`, 만렙 Lv.200. 전체/지역 동일 공식.

## 지역 / 카테고리
15. **지역: 17개 시도만 활성화.** 하위 지역(시/군/구)은 `Region` self-relation(parent)으로 구조만 열어둠. 음식점은 `primaryRegionId` 필수, `subRegionId` 선택(MVP 미사용).
16. **동일 음식점 식별 기준**: `(name, primaryRegionId)` 유니크. 같은 시도 내 동명 음식점은 1개로 합쳐 중복 등록 XP를 방지. 정교한 주소 기반 dedup은 향후.
17. **카테고리: DB seed로 관리(48개).** UI는 DB에서 읽어 렌더(하드코딩 금지). `type`별(상황/계절/인증/음식/가격)로 그룹핑.

## 랭킹
18. **읽기 경로는 `RankingCache` 우선(cron 갱신), 실시간 집계와 분리.** TOP 50 + 동점 tie-break(레벨→누적XP→최근30일XP). 캐시가 비면 실시간 쿼리로 폴백.
19. **이번 주 인기 맛집 점수**: 명세 가중치 중 **좋아요×1 + 저장×3**만 합산(이들은 타임스탬프가 있어 "이번 주" 계산 가능). 공유×5 / 방문인증×10 / 영상조회×0.1은 타임스탬프 이벤트 테이블 도입 후 합산(가중치 상수 `REACTION_WEIGHT`는 이미 정의). "이번 주"는 **월요일 00:00 기준**.
20. **검색의 "이번 주 인기순" 정렬**: 게시글 단위로 최근 7일 좋아요+저장 가중 점수로 재정렬.

## 사장님 / 수익화
21. **사장님 계정·홍보는 일반 사용자 앱 먼저.** `OwnerPromotion` 테이블 + 상세페이지에 "광고·사장님 홍보" 라벨로 **분리 노출**까지만. XP/랭킹과 완전 무관(코드 경로 자체가 분리).
22. **크리에이터 유료 맛집지도: 구현됨.** 판매자격(Lv.20 + 위치인증 30곳, 그중 영수증/메뉴 5곳) → 맛보기 5곳 선택 → 구매(PortOne V2 결제, webhook 서명검증) → 정산(수수료 30%)·출금. 프리미엄 구독은 미구현.

## 기타
23. **공유**: 공유 수(`shareCount`) 컬럼만 존재. 공유 카드 생성/리스트 공유 UI는 Phase 3.
24. **댓글/방문인증: 구현됨.** 댓글(1단계 답글, 좋아요, 고정, 욕설 필터), 방문인증(위치 GPS + 영수증/메뉴 AI). 위치인증이 XP 해제 게이트.
25. **프로덕션 빌드는 NTFS 필요**: 이 저장소가 위치한 `F:`는 **exFAT** 파일시스템이라 심볼릭링크/reparse-point를 지원하지 않음. `next build`가 모듈 realpath 해석 시 호출하는 `fs.readlink`가 exFAT에서 `EISDIR`로 실패함(webpack·Turbopack·Node 22/24 모두 동일). → **NTFS 경로(예: `C:\matzip`)로 복사 후 빌드**하면 성공(검증 완료: 12개 라우트 빌드 + `npm start` 정상). dev/smoke/tsc/lint 는 exFAT에서도 정상. (초기에 "비-ASCII 경로 문제"로 오진했으나 실제 원인은 exFAT.) 이 이슈 때문에 Next 를 빌드가 검증된 **15.5.19** 로 고정함.
