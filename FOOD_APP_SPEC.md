# 맛집 레벨업 소셜 앱 기획/구현 명세

## 0. 문서 목적

이 문서는 맛집 앱 MVP를 개발하기 위한 제품 기획서이자 구현 기준 문서다.
코딩은 별도 에이전트/개발자가 진행하더라도, 이 문서만 보고 앱의 핵심 구조를 이해하고 구현을 시작할 수 있어야 한다.

핵심은 단순한 맛집 검색 앱이 아니다.

> 사용자가 자신만의 맛집을 등록하고, 공유하고, 반응을 받으며, 전체/지역 레벨을 올리는 소셜 맛집 앱.

이 앱은 네이버지도/카카오맵/구글맵처럼 모든 정보를 제공하는 지도 앱이 아니라, 사람들이 "진짜 가보고 싶은 이유"를 기준으로 맛집을 발견하고, 자신의 맛집력을 성장시키는 앱이다.

---

## 1. 제품 한 줄 정의

> 내 맛집 지도를 키우고, 진짜 맛집을 잘 아는 사람으로 레벨업하는 소셜 맛집 앱.

조금 더 기능적으로 표현하면:

> 사용자가 맛집을 저장/등록/리뷰/공유하면 XP를 얻고, 전체 레벨과 지역별 레벨이 오르며, 랭킹에 진입할 수 있는 맛집 커뮤니티 앱.

## 2. 핵심 철학

### 2.1 진정성

이 앱의 가장 중요한 가치는 진정성이다.

블로그 체험단처럼 광고/협찬 때문에 무조건 좋게 써주는 리뷰앱이 되면 안 된다. 사장님은 자신의 가게를 진정성 있게 소개할 수 있고, 소비자는 직접 방문해서 솔직하게 리뷰할 수 있어야 한다.

원칙:

- 광고/홍보/쿠폰/캠페인은 일반 랭킹과 분리한다.
- 광고성 콘텐츠는 명확하게 표시한다.
- 사장님은 사용자 리뷰를 마음대로 삭제할 수 없다.
- 사용자의 솔직한 평가를 보호한다.
- 평점 경쟁보다 "어떤 상황에 가기 좋은 곳인가"를 강조한다.

### 2.2 상황 중심 검색

기존 지도앱은 보통 음식 종류, 별점, 거리 중심이다.
이 앱은 "왜 가고 싶은가"를 중심으로 한다.

예:

- 오늘 날씨 좋은데 야장 가능한 곳
- 비 오는 날 국물 먹기 좋은 곳
- 부모님 모시고 가기 좋은 오래된 맛집
- 데이트하기 좋은 분위기 맛집
- 돈 없을 때 가기 좋은 가성비 맛집
- 여름 계절메뉴가 좋은 곳
- 블루리본/미슐랭/노포 같은 인증 맛집

### 2.3 게임화

맛집 등록은 단순 저장이 아니라 성장 행동이다.

사용자는 맛집을 등록하고, 사진/영상/카테고리/리뷰를 채우고, 다른 사람에게 좋아요/저장/방문 반응을 받으면서 XP를 얻는다.

전체 레벨은 전국 활동을 합산해서 오른다.
지역 레벨은 해당 지역 맛집 활동만 따로 계산해서 오른다.

## 3. 핵심 사용자

### 3.1 일반 사용자

목표:

- 내 맛집을 저장한다.
- 내가 찾은 맛집을 공유한다.
- 다른 사람의 맛집을 보고 저장한다.
- 레벨과 랭킹을 올린다.

대표 심리:

- "나 이런 맛집 알아."
- "이 동네는 내가 잘 알아."
- "내가 올린 맛집이 좋아요 받으면 기분 좋다."
- "사람들이 내 맛집을 저장하면 인정받는 느낌이 든다."

### 3.2 맛집 큐레이터/크리에이터

목표:

- 자신만의 맛집 지도를 만든다.
- 특정 주제의 맛집 리스트를 운영한다.
- 팔로워를 모은다.
- 향후 유료 지도/유료 큐레이션으로 수익화할 수 있다.

예:

- 서울 야장 지도
- 을지로 노포 지도
- 강남 가성비 데이트 지도
- 아이랑 가기 좋은 식당 지도
- 부산 돼지국밥 지도

### 3.3 음식점 사장님

목표:

- 자신의 가게를 진정성 있게 소개한다.
- 쿠폰/이벤트/신메뉴/계절메뉴/야장 가능 여부를 알린다.
- 앱 사용자들이 방문하고 솔직한 리뷰를 남긴다.

주의:

- 사장님 콘텐츠는 일반 사용자 랭킹을 조작하면 안 된다.
- 사장님 홍보와 사용자 리뷰는 명확하게 분리한다.

## 5. 랭킹 구조

랭킹은 딱 3개만 운영한다.
랭킹이 너무 많으면 사용자가 헷갈리고 경쟁심이 분산된다.

### 5.1 전체 랭킹

기준:

- 전체 레벨 순
- 동점이면 전체 누적 XP 순
- 그래도 동점이면 최근 30일 활동 XP 순

표시:

- TOP 50
- 내 현재 순위
- 내 위/아래 근접 사용자

목적:

- 앱 전체에서 가장 영향력 있는 맛집 유저를 보여준다.

### 5.2 지역 랭킹

기준:

- 선택 지역의 지역 레벨 순
- 동점이면 해당 지역 누적 XP 순
- 그래도 동점이면 해당 지역 최근 30일 활동 XP 순

중요:

- 지역 랭킹은 전체 레벨을 그대로 쓰면 안 된다.
- 사용자가 서울 맛집을 많이 등록하면 서울 지역 레벨이 오른다.
- 사용자가 경남 맛집을 많이 등록하면 경남 지역 레벨이 오른다.
- 거주지가 어디인지는 지역 랭킹의 본질이 아니다.
- 해당 지역 맛집에 기여한 만큼 그 지역 랭킹에 오른다.

예:

- 전체 Lv.80
- 서울 Lv.35
- 경남 Lv.62
- 부산 Lv.12

이 사용자는 전체 랭킹에도 참여하고, 서울/경남/부산 지역 랭킹에도 각각 참여할 수 있다.

### 5.3 이번 주 인기 맛집 랭킹

기준:

- 이번 주 동안 음식점이 받은 반응 점수
- 좋아요, 저장, 조회, 공유, 방문 인증 등을 합산

권장 점수:

- 좋아요: 1점
- 저장: 3점
- 공유: 5점
- 방문 인증 리뷰: 10점
- 영상 조회: 0.1점

표시:

- TOP 50
- 전체/지역 필터 가능

목적:

- 지금 뜨는 맛집을 보여준다.
- 계절, 날씨, 야장, 신상, 이벤트 등 트렌드를 반영한다.

## 6. 레벨/XP 구조

### 6.1 레벨 종류

레벨은 2종류다.

1. 전체 레벨
2. 지역 레벨

전체 레벨:

- 모든 지역 활동 XP를 합산한다.
- 전체 랭킹 기준이다.
- 만렙은 Lv.200이다.

지역 레벨:

- 해당 지역 맛집 활동 XP만 합산한다.
- 지역 랭킹 기준이다.
- 지역별로 따로 계산한다.
- 각 지역 레벨도 만렙은 Lv.200이다.

예:

- 서울 맛집 등록: 전체 XP 증가 + 서울 지역 XP 증가
- 경남 맛집 등록: 전체 XP 증가 + 경남 지역 XP 증가
- 부산 맛집이 좋아요 받음: 전체 XP 증가 + 부산 지역 XP 증가

### 6.2 XP 지급 원칙

XP는 시원시원하게 지급하되, 레벨업 요구량은 점점 커져야 한다.
초반에는 빠르게 성장하고, 후반에는 꾸준한 활동과 좋은 반응이 필요해야 한다.

중요한 설계 방향:

- 단순 등록보다 반응이 더 중요하다.
- 좋아요보다 저장/방문 인증이 더 중요하다.
- 사진보다 영상이 더 높은 가치를 가진다.
- 카테고리와 정보 입력을 잘할수록 보너스를 준다.
- 무작정 많이 올리는 사람보다, 사람들이 실제로 저장하고 방문하게 만드는 사람이 유리해야 한다.

### 6.3 XP 지급표

#### 맛집 등록/작성 XP

| 행동 | XP |
|---|---:|
| 맛집 최초 등록 | 100 |
| 사진 1장 이상 추가 | 30 |
| 영상 1개 이상 추가 | 80 |
| 한줄평 작성 | 40 |
| 상세 리뷰 작성 | 80 |
| 카테고리 3개 이상 입력 | 40 |
| 가격대 입력 | 20 |
| 웨이팅 정보 입력 | 20 |
| 분위기/상황 태그 입력 | 30 |
| 재방문 의사 선택 | 20 |
| 메뉴 추천 입력 | 30 |

맛집 하나를 제대로 등록하면 약 250~450 XP를 얻도록 설계한다.

#### 반응 XP

| 행동 | XP |
|---|---:|
| 내 맛집이 좋아요 받음 | 10 |
| 내 맛집이 다른 사용자에게 저장됨 | 25 |
| 내 맛집이 공유됨 | 40 |
| 댓글 받음 | 10 |
| 다른 사용자가 "가봤어요" 인증 | 80 |
| 내 영상이 일정 조회수 도달 | 50 |

#### 장기 활동 XP

| 행동 | XP |
|---|---:|
| 하루 첫 맛집 등록 | 100 |
| 7일 연속 활동 | 500 |
| 한 지역 맛집 10곳 등록 | 800 |
| 한 카테고리 맛집 10곳 등록 | 800 |
| 신고 없이 30일 활동 | 1000 |

### 6.4 XP 상한/어뷰징 방지

초기 MVP에서도 최소한 다음 제한은 고려해야 한다.

- 동일 음식점 중복 등록 XP 없음
- 하루 맛집 등록 기본 XP는 최대 10건까지만 지급
- 좋아요 XP는 하루 최대치 설정
- 같은 사용자에게 반복적으로 받는 좋아요는 감쇠 적용
- 새 계정/비정상 계정의 좋아요는 XP 반영을 늦게 하거나 낮게 반영
- 사장님 홍보/광고/쿠폰 참여는 일반 랭킹 XP에 직접 반영하지 않음
- 신고 누적 콘텐츠는 XP 지급 보류

좋아요 어뷰징 감쇠 예:

- 같은 사용자 A가 사용자 B에게 하루 10개 좋아요를 눌러도 XP는 최대 3개까지만 인정

### 6.5 레벨 공식

만렙은 Lv.200이다.

초반은 빠르게, 후반은 느리게 가야 한다.
구현자는 다음 공식을 기본값으로 사용한다.

```text
requiredXpForNextLevel(level) = 300 + floor(level ^ 2.15 * 90)
```

설명:

- level은 현재 레벨이다.
- Lv.1에서 Lv.2로 갈 때 필요한 XP는 낮다.
- 레벨이 올라갈수록 필요 XP가 비선형으로 증가한다.
- Lv.200 이후는 더 이상 레벨이 오르지 않는다.

권장:

- 화면에는 공식이나 총 필요 XP를 복잡하게 보여주지 않는다.
- 사용자에게는 현재 레벨, 다음 레벨까지 남은 XP, 진행바만 보여준다.

레벨 구간 감각:

| 구간 | 성장 감각 |
|---|---|
| Lv.1~20 | 빠르게 성장, 초반 재미 |
| Lv.21~50 | 꾸준히 등록하면 도달 |
| Lv.51~100 | 반응 좋은 맛집이 필요 |
| Lv.101~150 | 지역 랭킹권 유저 |
| Lv.151~200 | 최상위 유저 |

### 6.6 레벨 계산 구현 방식

DB에는 최소 다음 값을 저장한다.

- user.total_xp
- user.total_level
- user_region_stats.region_id
- user_region_stats.region_xp
- user_region_stats.region_level

XP 이벤트가 발생할 때:

1. XP 이벤트 로그를 저장한다.
2. 전체 XP에 더한다.
3. 해당 맛집 지역의 지역 XP에 더한다.
4. 전체 레벨과 지역 레벨을 다시 계산한다.
5. 랭킹 캐시를 갱신하거나 비동기 작업으로 미룬다.

XP 이벤트는 반드시 로그로 남긴다.
나중에 어뷰징/버그/밸런스 조정에 필요하다.

## 7. 지역 구조

### 7.1 기본 지역

초기 지역 랭킹은 대한민국 17개 시도 단위로 시작한다.

- 서울
- 경기
- 인천
- 부산
- 대구
- 대전
- 광주
- 울산
- 세종
- 강원
- 충북
- 충남
- 전북
- 전남
- 경북
- 경남
- 제주

초기부터 시/군/구/동 단위 랭킹을 전부 만들지 않는다.
너무 세세하면 랭킹이 비어 보이고 경쟁심이 약해진다.

### 7.2 하위 지역 확장

사용자가 많아진 지역만 하위 지역 랭킹을 추가로 연다.

오픈 조건 예:

- 해당 지역 맛집 300개 이상
- 해당 지역 활동 유저 100명 이상
- 최근 30일 맛집 등록 100건 이상

예:

- 서울: 강남/서초, 마포/홍대/연남, 성수/성동, 종로/을지로, 잠실/송파, 용산/이태원
- 경기: 수원, 성남/분당, 고양, 용인, 부천, 안양
- 경남: 창원, 진주, 사천, 거제, 김해, 통영
- 부산: 해운대, 서면, 광안리, 남포

### 7.3 음식점의 지역 귀속

각 음식점은 반드시 하나의 primary region을 가진다.

예:

- 서울 성동구 성수동 음식점: primary region = 서울
- 경남 진주시 음식점: primary region = 경남

하위 지역이 오픈된 경우:

- primary region = 서울
- sub region = 성수/성동

XP는 기본적으로 primary region에 쌓인다.
하위 지역 랭킹을 운영할 경우 sub region XP도 별도로 쌓을 수 있다.

MVP에서는 primary region만 구현해도 된다.

## 8. 카테고리/태그 구조

카테고리는 음식 종류보다 "가고 싶은 이유"를 중심으로 설계한다.

### 8.1 기본 카테고리

상황/욕구 기반:

- 가성비
- 분위기

## 13. 수익화 구조

### 13.1 일반 사용자 구독

무료:

- 맛집 등록 가능
- 기본 저장 가능
- 기본 공유 가능
- 광고 노출
- 공유 카드 워터마크
- 일부 고급 통계 제한

프리미엄:

- 광고 제거
- 공유 카드 워터마크 제거
- 고급 공유 템플릿
- 맛집 리스트 무제한 생성
- 비공개 지도
- 고급 필터/통계

가격 초안:

- 월 1,100원: 광고 제거
- 월 2,200원: 광고 제거 + 워터마크 제거 + 고급 공유/리스트 기능

초기에는 너무 많은 요금제를 보여주지 않는다.
MVP에서는 `무료`와 `프리미엄 월 2,200원` 정도가 가장 단순하다.

### 13.2 사장님 광고/홍보

사장님 기능은 "좋게 써주는 광고"가 아니라 "진정성 있는 가게 소개"여야 한다.

가능 상품:

- 가게 소개 페이지 강화
- 신메뉴/계절메뉴 노출
- 야장 가능 여부 노출
- 쿠폰/이벤트 등록
- 지역 광고 탭 노출

주의:

- 사장님 유료 노출은 일반 랭킹에 직접 영향을 주지 않는다.
- 광고/홍보 표시는 명확히 한다.
- 사용자 리뷰는 조작할 수 없다.

### 13.3 크리에이터 지도 수익화

장기 핵심 수익 모델.

구조:

- 크리에이터가 맛집 지도를 만든다.
- 사용자가 해당 지도를 팔로우하거나 유료 구매/구독한다.
- 앱은 수수료를 가져간다.

예:

- 지도 단건 구매: 1,100~3,300원
- 월 구독: 1,100원
- 수익 배분: 앱 30%, 크리에이터 70% 또는 앱 50%, 크리에이터 50%

MVP에서는 결제까지 구현하지 않아도 된다.
단, 데이터 모델은 나중에 유료 지도로 확장 가능하게 설계한다.

## 14. 사장님 캠페인 방향

기존 블로그 체험단식 협찬 리뷰는 지양한다.

대신 다음 구조를 목표로 한다.

1. 사장님이 자신의 가게를 소개한다.
2. 사용자는 그 소개를 보고 자발적으로 방문한다.
3. 사용자는 솔직한 리뷰를 남긴다.
4. 리뷰는 좋을 수도, 별로일 수도 있다.
5. 사장님은 리뷰를 삭제하거나 점수를 조작할 수 없다.

캠페인/쿠폰이 있는 경우:

- 사용자가 혜택을 받았는지 표시한다.
- 혜택을 받았어도 리뷰 내용은 자유롭게 작성한다.
- 혜택 제공 리뷰는 일반 리뷰와 구분 표시한다.

## 18. 코딩 시 반드시 지킬 것

### 18.1 XP 로직은 한 곳에 모은다

XP 지급 로직이 화면/컨트롤러/API 곳곳에 흩어지면 유지보수가 어렵다.

반드시 별도 서비스로 만든다.

예:

```text
XpService.awardXp(userId, regionId, sourceType, sourceId)
LevelService.calculateLevel(totalXp)
RankingService.refresh()
```

### 18.2 XP 이벤트는 멱등성을 고려한다

같은 이벤트가 중복 실행되면 XP가 두 번 들어가면 안 된다.

예:

- 같은 post_created 이벤트는 한 번만
- 같은 user가 같은 post에 like한 경우 한 번만
- unlike 후 다시 like했을 때 XP를 다시 줄지 정책 필요

MVP 정책:

- 최초 좋아요에만 XP 지급
- 좋아요 취소 후 재좋아요는 XP 재지급 없음

### 18.3 지역 레벨 업데이트 누락 금지

음식점 관련 XP는 전체 XP와 지역 XP를 동시에 업데이트해야 한다.

예:

- 서울 맛집 좋아요 받음
- 작성자 total_xp 증가
- 작성자 서울 region_xp 증가

### 18.4 광고/사장님 콘텐츠는 랭킹 XP에 직접 영향 없음

사장님이 돈을 냈다고 랭킹에 오르면 앱 신뢰가 무너진다.

광고 노출:

- 별도 영역에서 노출
- 광고 표시
- 일반 랭킹과 분리

### 18.5 카테고리는 하드코딩하지 않는다

카테고리는 DB seed로 관리한다.
나중에 추가/수정 가능해야 한다.

### 18.6 업로드 파일은 확장 가능하게 설계한다

MVP에서는 이미지 중심이어도, 데이터 모델은 video를 지원해야 한다.
릴스형 피드가 중요하기 때문이다.

### 18.7 정렬/랭킹은 성능 고려

초기에는 단순 쿼리로 가능하지만, TOP 50 랭킹은 캐시 테이블이나 주기적 재계산 구조로 확장 가능해야 한다.

## 21. 현재 합의된 핵심 결정

- 필터 앱 방향보다 맛집 소셜/게임화 앱 방향으로 간다.
- 랭킹은 맛집 랭킹 중심이 아니라 사용자 랭킹 중심이다.
- 랭킹은 3개만 둔다.
- 전체 랭킹은 전체 레벨 순이다.
- 지역 랭킹은 지역별 레벨 순이다.
- 이번 주 인기 맛집 랭킹은 음식점 반응 기준이다.
- 전체 레벨과 지역 레벨은 별도로 운영한다.
- 만렙은 Lv.200이다.
- 칭호 시스템은 넣지 않는다.
- 지역은 처음엔 17개 시도 단위로 시작한다.
- 사용자가 많아진 지역만 시/구/핫플 단위로 확장한다.
- XP는 시원하게 지급하되, 후반 레벨업은 오래 걸리게 한다.
- 카테고리는 상황/날씨/계절/분위기/가성비/노포/야장/인증 맛집 등 사람 심리를 건드리는 방향으로 설계한다.
- 사장님 홍보는 가능하지만 일반 랭킹과 분리한다.
- 블로그 체험단 같은 가짜 긍정 리뷰 구조는 지양한다.
- 크리에이터 맛집 지도는 장기 수익화 후보로 유지한다.
